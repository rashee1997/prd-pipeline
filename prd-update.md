---
description: "PRD Update — Amend PRD mid-pipeline and cascade changes to spec, plan, and tasks. Triggered by implementation blockers (/prd-implement --existing) or manual intent/requirement changes at any stage after discovery."
argument-hint: "<feature-folder> [\"description of change\"] | --existing \"<structured-prompt>\""
allowed-tools: mcp__semble, mcp__serena, mcp__octocode, mcp__context7, Bash
---

<command name="/prd-update">

  <shared_rules>_shared.md — base execution, semble-first, UNVERIFIED protocol, and MCP fallback apply to this command.</shared_rules>

  <execution><!-- base rules in _shared.md -->
    <prefer_mcp_for_research>true</prefer_mcp_for_research>
    <do_not_implement_code>true</do_not_implement_code>
  </execution>

  <system>
    <role>Senior spec architect and cascade coordinator</role>
    <principle>Understand the change → research impact → cascade minimum-invasively → invalidate stale gates</principle>
    <mode>research-backed cascade update — fewest artifacts changed, full traceability preserved</mode>
    <rules>
      <item>Do not implement product code.</item>
      <item>Do not modify completed [x] tasks unless the change directly contradicts their committed implementation.</item>
      <item>Always invalidate tasks/validation.md when tasks are added, modified, or removed.</item>
      <item>Update artifacts upstream → downstream: discovery (if needed) → prd → spec → plan → tasks.</item>
      <item>Delta updates only — never rewrite unchanged sections.</item>
      <item>Every new symbol, path, or field introduced must pass CoVe verification before entering any updated artifact.</item>
      <item priority="critical">mcp__semble is MANDATORY for all update research — call mcp__semble__search before any other MCP tool.</item>
    </rules>
  </system>

  <input>
    <source>{$ARGUMENTS}</source>
    <accepted_forms>
      <form id="impl-blocker">--existing "{structured-prompt-from-prd-implement}"</form>
      <form id="manual">{feature-folder-path} "{free-form description of what changed}"</form>
      <form id="folder-only">{feature-folder-path}  ← will ask for description interactively</form>
    </accepted_forms>
    <output>Updated artifacts in-place + tasks/validation.md marked STALE if tasks changed</output>
  </input>

  <input_gate>
    <check>Either --existing with a structured prompt OR a feature folder path must be in $ARGUMENTS.</check>
    <if_missing>Stop. Print: "❌ Usage:\n  /prd-update docs/prd/{feature}/ \"what changed\"\n  /prd-update --existing \"{prompt from prd-implement}\"" Do not proceed.</if_missing>
    <check id="prd-exists" condition="mode=manual or folder-only">Feature folder must contain prd.md.</check>
    <if_missing_prd>Stop. Print: "❌ prd.md not found at {path}. Run /prd-write first."</if_missing_prd>
  </input_gate>

  <structured_prompt_spec condition="mode=impl-blocker">
    The --existing value is a pipe-delimited string generated by /prd-implement:
      TASK:{task-id} | BLOCKER:{type} | ISSUE:{description} | DISCOVERED:{actual-vs-expected} | AFFECTED_FILES:{comma-separated-paths} | FEATURE:{feature-folder-path}

    BLOCKER types:
      SPEC_WRONG        — spec names a symbol/route/field that does not exist in live code
      API_CHANGED       — external library API changed since the spec was written
      MISSING_DEP       — a dependency the task needs is absent from the manifest
      SCHEMA_MISMATCH   — a schema column/field name in the task contradicts the actual schema
      ASSUMPTION_INVALID — a task assumption (file exists, table has column, etc.) is false
  </structured_prompt_spec>

  <flow>

    <phase id="1" name="parse-and-orient">
      <task>Parse input, detect mode, and read the current state of all pipeline artifacts.</task>

      <if condition="--existing flag present">
        <steps>
          <step>Parse the pipe-delimited prompt: extract TASK, BLOCKER, ISSUE, DISCOVERED, AFFECTED_FILES, FEATURE.</step>
          <step>Set feature_folder = FEATURE value.</step>
          <step>Set change_type = IMPLEMENTATION_BLOCKER.</step>
          <step>Print: "🔍 Blocker context: Task {TASK}, type {BLOCKER} — {ISSUE}"</step>
        </steps>
      </if>

      <if condition="manual or folder-only">
        <steps>
          <step>Set feature_folder from first positional argument.</step>
          <step condition="description provided">Set change_description from second argument.</step>
          <step condition="no description">Ask one question: "What changed or what do you want to update? (requirement change, wrong spec assumption, new information, or intent change)"</step>
          <step>Infer change_type from description: INTENT_CHANGE | REQUIREMENT_ADD | CORRECTION.</step>
        </steps>
      </if>

      <read>
        {feature_folder}/prd.md
        {feature_folder}/spec.md
        {feature_folder}/plan.md          (if exists)
        {feature_folder}/tasks/index.md   (if exists)
      </read>

      <extract>
        <item>Current pipeline stage (which artifacts exist)</item>
        <item>Completed tasks — [x] rows in index.md</item>
        <item>Pending tasks — [ ] rows in index.md</item>
        <item>Feature name and slug</item>
        <item>PRD requirement IDs</item>
        <item>Spec section headings</item>
      </extract>
    </phase>

    <phase id="2" name="research-impact">
      <task>Research the change against the live codebase before deciding what to update. Evidence before action.</task>

      <semantic-discovery>
        <step tool="mcp__semble__search" required="true">
          Search for the affected area using a natural-language query derived from the issue or change description. Run 2-3 queries covering different facets of the change.
        </step>
        <step tool="mcp__semble__find_related" required="true">
          For the top semble result, find related code adjacent to the affected area.
        </step>
      </semantic-discovery>

      <symbol-verification condition="change_type=IMPLEMENTATION_BLOCKER or CORRECTION">
        <step tool="mcp__serena__find_symbol">Re-verify every symbol, path, and field named in the issue/change against the live codebase.</step>
        <step tool="mcp__octocode__localGetFileContent">Read affected files to confirm their actual current state.</step>
      </symbol-verification>

      <library-verification condition="BLOCKER=API_CHANGED or change touches external library usage">
        <step tool="mcp__context7__query-docs">Confirm the current external API shape for the library in question.</step>
        <step tool="mcp__octocode__packageSearch">Compare current published version vs spec-pinned version.</step>
      </library-verification>

      <schema-verification condition="BLOCKER=SCHEMA_MISMATCH or change affects DB schema">
        <step tool="mcp__octocode__localGetFileContent">Read the schema file (prisma/schema.prisma or equivalent) to confirm actual field names and types.</step>
      </schema-verification>

      <research-conclusion required="true">
        State findings explicitly before proceeding — do not skip this:
        - "Spec says {X}. Live code shows {Y}. Required change: {Z}."
        - "External API changed: {old signature} → {new signature} at version {V}."
        - "New requirement is {X}. Current spec covers {Y}. Gap: {Z}."
        This conclusion drives cascade-depth classification in phase 3.
      </research-conclusion>
    </phase>

    <phase id="3" name="classify-cascade-depth">
      <task>Determine which artifacts need updating and in what order. Default to the shallowest level that covers the change.</task>

      <levels>
        <level id="TASKS_ONLY">
          Triggers: change affects how tasks are structured but not requirements or spec content.
          Examples: a task's file path is wrong, acceptance command needs updating, task isolation issue.
          Updates: tasks/index.md + affected TASK-*.md files only.
          Skips: prd.md, spec.md, plan.md.
        </level>

        <level id="PLAN_AND_TASKS">
          Triggers: change adds, removes, or reorders implementation steps but not requirements or spec.
          Examples: a missing install task, wrong layer assignment, additional migration task needed.
          Updates: plan.md + tasks.
          Skips: prd.md, spec.md.
        </level>

        <level id="SPEC_CASCADE">
          Triggers: change corrects a technical spec assumption — API signature, schema field, component name, or file path. Does not change what the feature does.
          Examples: SPEC_WRONG, API_CHANGED, SCHEMA_MISMATCH, ASSUMPTION_INVALID blockers.
          Updates: spec.md (affected section only) → plan.md (if steps change) → tasks (affected tasks).
          Skips: prd.md unless a functional requirement changes.
        </level>

        <level id="FULL_CASCADE">
          Triggers: a functional requirement is added, removed, or meaningfully changed; user intent changed.
          Examples: INTENT_CHANGE, REQUIREMENT_ADD, removing a feature that has pending tasks.
          Updates: prd.md → spec.md → plan.md → tasks (full cascade, minimum invasive).
        </level>

        <level id="DISCOVERY_ADDENDUM">
          Triggers: new external research finding that does not change requirements or spec.
          Examples: discovered a related third-party integration to be aware of.
          Updates: append findings section to discovery.md only.
        </level>
      </levels>

      <routing-rules>
        <rule>A single wrong API name or field name → SPEC_CASCADE, not FULL_CASCADE.</rule>
        <rule>A changed user-facing requirement or removed feature → FULL_CASCADE.</rule>
        <rule>A missing install task or wrong task order → PLAN_AND_TASKS.</rule>
        <rule>A single wrong file path in a task → TASKS_ONLY.</rule>
        <rule>If in doubt: prefer SPEC_CASCADE over FULL_CASCADE to minimize rewrite scope.</rule>
      </routing-rules>

      <announce>
        Print: "📊 Cascade: {LEVEL} — updating: {list of artifacts}. Skipping: {list skipped}."
      </announce>
    </phase>

    <phase id="4" name="update-plan-preview">
      <task>Show exactly what will change before executing any updates.</task>

      <show>
        For each artifact to update:
        ├─ {artifact path}
        │    Section: {affected section name}
        │    Change: {one-line description}

        Tasks:
        ├─ ADD: {new task IDs + one-line description each, or none}
        ├─ MODIFY: {task ID + what changes, or none}
        ├─ SUPERSEDE: {task ID + reason — only if feature removed, or none}
        └─ SKIP (completed): {task IDs not touched}
      </show>

      <confirmation condition="cascade=FULL_CASCADE or tasks_to_supersede > 0">
        Print the full update plan and ask: "Proceed with these updates? (y/n)"
        If no: stop. If yes: proceed to phase 5.
      </confirmation>

      <auto-proceed condition="cascade=TASKS_ONLY or PLAN_AND_TASKS or SPEC_CASCADE">
        No confirmation needed — proceed directly.
      </auto-proceed>
    </phase>

    <phase id="5" name="cove-checkpoint" condition="new symbols, paths, or fields introduced by the change">
      <purpose>Prevent hallucinated names from entering updated artifacts.</purpose>
      <steps>
        <step>Extract every new symbol name, file path, API route, schema field, and component name introduced by the planned changes (not existing ones — only net-new).</step>
        <step tool="mcp__semble__search">Verify each new item via semble.</step>
        <step tool="mcp__serena__find_symbol" condition="semble misses">Re-verify misses via serena.</step>
        <step>Confirmed items: mark CONFIRMED (file:line). Unresolved items: mark [UNVERIFIED: name].</step>
        <step>Do not write [UNVERIFIED] items into any updated artifact. Either resolve them first, or note the gap explicitly and ask the user for clarification.</step>
      </steps>
    </phase>

    <phase id="6" name="execute-cascade">
      <task>Update artifacts in upstream → downstream order. Delta only — write only changed sections.</task>

      <update id="discovery" condition="cascade=DISCOVERY_ADDENDUM">
        Append a new `## Update — {dd-mm-yyyy}: {one-line summary}` section to discovery.md.
        Do not rewrite any existing content.
      </update>

      <update id="prd" condition="cascade=FULL_CASCADE">
        <rules>
          <rule>Update only the requirement(s) that changed. Leave all other sections unchanged.</rule>
          <rule>Adding a requirement: append to the correct requirements section with a new requirement ID (REQ-N+1).</rule>
          <rule>Removing a requirement: mark `~~REQ-N~~ REMOVED — {reason}` in place (preserves traceability). Do not delete.</rule>
          <rule>Modifying a requirement: update only that requirement's text and acceptance criteria. Leave adjacent requirements untouched.</rule>
          <rule>Update PRD version header (bump minor version) and date.</rule>
        </rules>
      </update>

      <update id="spec" condition="cascade=SPEC_CASCADE or FULL_CASCADE">
        <rules>
          <rule>Update only the section(s) affected: API definition, schema diff, component spec, or integration section.</rule>
          <rule>Wrong field names or API signatures: correct the specific definition only.</rule>
          <rule>Schema change: update only the schema diff section with corrected field names/types.</rule>
          <rule>API change: update only the route/method/parameter definition. Do not touch unrelated routes.</rule>
          <rule>Do not rewrite blast-radius, compatibility, or testing sections unless they directly reference the corrected item.</rule>
          <rule>Update spec version header (bump patch) and date.</rule>
        </rules>
      </update>

      <update id="plan" condition="cascade=PLAN_AND_TASKS or SPEC_CASCADE or FULL_CASCADE">
        <rules>
          <rule>Update only the steps affected by the change.</rule>
          <rule>New steps needed: add to the dependency graph at the correct layer. Assign IDs that don't conflict with existing steps.</rule>
          <rule>Steps no longer needed: mark REMOVED with reason. Do not delete (preserves traceability).</rule>
          <rule>Order changes: update only the affected dependency edges.</rule>
          <rule>Keep plan.md lean — no per-step build bullets (those go in task files).</rule>
        </rules>
      </update>

      <update id="tasks" condition="any cascade level">

        <completed-tasks-policy>
          <rule>Do NOT rewrite or modify any task marked [x] in index.md.</rule>
          <rule>Exception: if the change directly contradicts the committed implementation, add a warning note below the [x] row:
            `  > ⚠️ **Spec updated {date}:** {one sentence: what changed and what code may need amendment}.`
            Do not touch the task file itself — the code change is out of scope for this command.
          </rule>
        </completed-tasks-policy>

        <pending-tasks>
          <rule>For each pending task ([ ]) that references the changed spec section: update only the affected context block and/or the relevant step(s) in that task file. Leave all other task content unchanged.</rule>
          <rule>Task made obsolete by the change: mark `[~] SUPERSEDED — {reason}` in index.md. Do not delete the task file.</rule>
          <rule>New tasks required: create new TASK-*.md files following the standard prd-tasks format (role, context, task, acceptance, commit). Add to index.md at the correct layer position.</rule>
        </pending-tasks>

        <index-update>
          <rule>Update the progress total (N tasks) if tasks were added or superseded.</rule>
          <rule>Append at the bottom of index.md:
            `## Update — {dd-mm-yyyy}: {one-line summary of what changed}`
            List tasks added, modified, and superseded (one line each).
          </rule>
        </index-update>

      </update>
    </phase>

    <phase id="7" name="invalidate-validation">
      <task>Mark existing validation as stale when tasks changed, so prd-implement cannot proceed without re-validating.</task>

      <if condition="tasks/validation.md exists AND tasks were added, modified, or superseded">
        <steps>
          <step>Change the `status:` field in validation.md frontmatter to `STALE`.</step>
          <step>Prepend to the top of validation.md (after frontmatter):
            ```
            > ⚠️ **Validation stale** — PRD updated {dd-mm-yyyy}. Re-run `/prd-validate {tasks-path}` before implementing.
            > Change: {one-line summary}
            ```
          </step>
        </steps>
      </if>

      <if condition="no validation.md exists or tasks were not changed">
        Skip — no invalidation needed.
      </if>
    </phase>

    <phase id="8" name="output">
      <template>
        ✅ PRD Updated — {feature-name}

        **Change type:** {IMPLEMENTATION_BLOCKER|INTENT_CHANGE|REQUIREMENT_ADD|CORRECTION}
        **Cascade depth:** {LEVEL}

        **Artifacts updated:**
        - {path}: {one-line summary of what changed}
        ...

        **Tasks:**
        - Added:     {IDs or none}
        - Modified:  {IDs or none}
        - Superseded: {IDs or none}
        - Skipped (completed [x]): {IDs or none}

        **Validation:** {⚠️ STALE — must re-run | ✅ unchanged — not required}

        **Next step:**
        {if tasks changed}
        /prd-validate {tasks-folder}/index.md

        {if tasks unchanged and validation was VALIDATED}
        /prd-implement {next-pending-task-path}
      </template>
    </phase>

  </flow>

  <control>
    <priority>minimum-invasive cascade over completeness theater</priority>
    <failure>if research phase cannot confirm what changed, ask one specific clarifying question before proceeding — do not guess</failure>

    <hard_rules>
      <rule>Do not implement product code.</rule>
      <rule>Do not rewrite completed [x] task files.</rule>
      <rule>Do not rewrite unchanged sections of any artifact.</rule>
      <rule>Do not introduce [UNVERIFIED] items into updated artifacts.</rule>
      <rule>Always invalidate validation.md when tasks are modified.</rule>
      <rule>Always update artifacts upstream → downstream.</rule>
      <rule>mcp__semble is mandatory — call mcp__semble__search before any other MCP tool.</rule>
    </hard_rules>
  </control>

  <final>
    <if condition="validation-stale">
      ## ⚠️ Re-validation Required

      Tasks were modified. Run validation before implementing:
      `/prd-validate {tasks-folder}/index.md`

      Then resume:
      `/prd-implement {tasks-folder}/{next-pending-task}.md`
    </if>

    <if condition="no-task-changes">
      ## ✅ Artifacts Updated — No Re-validation Needed

      Continue implementation:
      `/prd-implement {tasks-folder}/{next-pending-task}.md`
    </if>
  </final>

</command>
